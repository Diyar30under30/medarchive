# MedArchive — Design & Architecture Decisions

**Date:** 2026-06-27
**Source spec:** `MedArchive_ClaudeCode_BuildPrompt.md` (the locked hackathon brief — authoritative for requirements).
**This document:** the decisions and adaptations layered on top of the brief, validated via brainstorming.

---

## 1. Context

MedArchive turns a messy archive of clinic price lists (text PDF, scanned PDF, XLSX, DOCX with tracked changes) into a clean, queryable, verified database of "who provides which medical service and at what price," normalized to a single canonical service directory.

Judged rubric (optimize in this order):

| Weight | Criterion |
|---|---|
| 30% | Data extraction quality (PDF/OCR/XLSX/DOCX) |
| 25% | Normalization & matching |
| 20% | Validation & verification |
| 15% | API & search (OpenAPI) |
| 10% | Operator UX |

The strategic core (45% combined) is **name normalization** — mapping raw strings (`ОАК`, `CBC`, `Кл. ан. крови`) to one canonical `Общий анализ крови`, with a matcher that **learns from operator confirmations**.

## 2. Environment reality (detected 2026-06-27)

- Docker: **not installed**. Postgres/pgvector: **not installed**. Tesseract: **not installed**.
- Python: **3.14.5** (newer than the brief's 3.11 — `torch`/`sentence-transformers` wheels may be missing on host).
- Node 26.1.0, git 2.54: present.

The host cannot run the brief's locked stack today. This drives the central architectural decision below.

## 3. Central decision — two-profile runtime

**Build the canonical Docker stack exactly per the brief AND make the backend stack-agnostic so it also runs locally without Docker.** Approved during brainstorming.

- **Judge-facing path (brief-compliant):** `docker-compose.yml` → Python 3.11, Postgres 16 + pgvector, Tesseract (`rus+kaz+eng`), MiniLM model baked into the image. Real `docker compose up`. Offline at judging time.
- **Dev path on this machine (no Docker):** SQLAlchemy is DB-agnostic → a **SQLite dev mode** runs pipeline + API + frontend on the host now. Matching **degrades gracefully**:
  - RapidFuzz lexical (pure-Python) — always available.
  - sentence-transformers semantic layer — active when installed (always in Docker, optional on host).
  - OCR — feature-flagged; skipped with a logged warning if Tesseract absent, full in Docker.

Selection is config-driven (`config.py` + `.env`): `DATABASE_URL`, `ENABLE_EMBEDDINGS`, `ENABLE_OCR`.

**Rationale:** honors the brief (Docker stack present, correct, offline) while keeping the system *runnable at every step* on the actual machine, instead of a repo that can't start until three system packages are installed.

## 4. Components & boundaries

```
medarchive/
  docker-compose.yml, .env.example, README.md
  data/ reference/ incoming/ raw_store/
  backend/
    app/
      main.py config.py
      db/            engine, session, base (Postgres OR SQLite via DATABASE_URL)
      models/        SQLAlchemy entities (Service, Partner, PriceDocument, PriceItem, Job, MatchReviewItem)
      schemas/       Pydantic
      ingestion/     unzip, format detect (ext + magic + text-layer probe), dispatcher, job tracking
      extractors/    base.py + registry; pdf_text, pdf_scan_ocr, xlsx, docx_tracked
      normalization/ preprocess, abbreviations, matcher, embeddings, reference_loader
      validation/    rules, currency
      versioning/    history (supersede prior active PriceItem)
      api/           routers per resource
      workers/       BackgroundTasks + DB-backed job table
    scripts/         generate_reference, load_reference, generate_fixtures, ingest (CLI), evaluate_matching
    tests/
  frontend/          React + Vite + TS + Tailwind (Search, Service, Partner, Admin/{Upload,Queue,Dashboard})
```

**Strategy/registry pattern** for extractors (`EXTRACTOR_REGISTRY[file_format] -> Extractor`) and matchers — adding a format/source touches no core code (non-functional requirement).

Each extractor returns a uniform `list[ExtractedRow]`. One failing file never halts the batch (document `parse_status=error` + `parse_log`, continue).

## 5. Data model

Exactly the brief's entities (Section 6 of the brief), UUID PKs, `created_at`/`updated_at`. Internal `service_id` (UUID5 of `Специальность|Code|Name_ru`) generated by the loader because no source column is a usable key. `Name_ru` repeats across specialties → never silently pick one; disambiguate by specialty or send to review. `embedding`: pgvector column (PG) / serialized float array (SQLite).

## 6. Matching engine (the crux)

Standalone, unit-tested `normalization/matcher.py`:
`match(raw_name, *, specialty_hint=None, top_k=5) -> list[Candidate{service_id, score, method}]`

Pipeline: preprocess/canonicalize (lowercase, homoglyph unify, collapse punctuation, expand abbreviation map, strip noise qualifiers but keep as tie-break features) → exact/synonym (1.0) → RapidFuzz lexical top-k → semantic top-k (when embeddings available) → fusion `0.45*lex + 0.55*sem` (config) → specialty boost → thresholds (`≥0.85` auto-match/verified; `0.65–0.85` needs_review with ranked candidates; `<0.65` unmatched) → **learning loop**: operator confirm appends canonical raw form to `Service.synonyms` so similar future rows auto-match.

`evaluate_matching.py` prints precision / auto-match rate over fixtures (target ≥70%). Store `match_method` + candidate scores for explainability.

## 7. Reference + fixtures (host has no real XLSX)

- `generate_reference.py` → realistic synthetic `data/reference/Справочник услуг.xlsx`: services across multiple specialties, RU/KK names, repeated `Name_ru` across specialties, no clean key. Columns `ID | Специальность | Code | Name_ru | TarificatrCode`.
- `load_reference.py` accepts **that synthetic file OR a real XLSX/JSON** with the same column mapping, idempotently (UUID5, category=Специальность, source_code=Code, tariff_code=TarificatrCode, synonyms=[], compute embedding when available, upsert).
- `generate_fixtures.py` → `data/incoming/sample_archive.zip`: ≥8 price lists across all 4 formats with deliberate messiness (abbreviations, Latin variants, typos, qualifiers, Kazakh names; resident/non-resident; USD/RUB rows; the anomalies: non-resident<resident, >50% jump vs earlier file, duplicate, empty name, future date; ≥1 scan-style PDF; ≥1 DOCX with tracked changes). Saves a ground-truth manifest for tests.

## 8. Validation, currency, versioning

All Section-9 rules implemented (price>0, non-resident≥resident, empty-name skip, future-date, duplicate dedup, >50% anomaly, currency convert keeping `price_original`+`currency_original`, no-data→error). Configurable per-date rate table (static fallback) for offline. Prices versioned, never overwritten: newer-dated doc supersedes prior active `PriceItem` (`superseded_by`), history kept indefinitely.

## 9. API & search

All Section-11 endpoints, Pydantic schemas, FastAPI auto OpenAPI at `/docs` + `/openapi.json`. Search via Postgres FTS + trigram (PG) / `LIKE` + RapidFuzz (SQLite dev), ≤~1s on demo dataset.

## 10. Frontend

React+Vite+TS+Tailwind, built with the `ui-ux-pro-max` skill. Public: Search (autocomplete, partner cards, sort/filter), Service (price compare), Partner (full price list + freshness). Operator `/admin`: Upload & live status, Verification queue (source fragment | suggested match + ranked alternatives with scores | Confirm/Correct/Reject, keyboard-first), Dashboard (docs processed, **% auto-normalized** headline, queue depth, errors, matches-by-method via Recharts). Bilingual labels (RU primary).

## 11. Build sequence (multi-turn, runnable + committed each phase)

A0 scaffold → A1 reference → A2 fixtures → A3 extractors → A4 matcher → A5 validation/currency/versioning → A6 API → A7 frontend → A8 tests/report/README. Part B only after Part A passes its Definition of Done (brief Section 17).

## 12. Non-goals / YAGNI for now

Celery+Redis, MeiliSearch, Leaflet map, ICD enrichment, bulk export, БИН merge tool, RU/KK UI toggle — all Part B, deferred until Part A is green.
